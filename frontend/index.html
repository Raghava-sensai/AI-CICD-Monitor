📖 Understanding the Monitor

          
          
Deployment Statuses

          
    
             
  • Healthy The code successfully compiled, the container started, and the application responded to the health check ping.
    • 
             
  • Warnings The container successfully built and is running, but the application health check failed. This usually means the application is running, but listening on an unexpected port inside the container, or it took too long to boot.
    • 
             
  • Failed The pipeline crashed entirely. This means the code couldn't be cloned, the Dockerfile failed to build, or the container crashed immediately on startup.
    • 
          


          
Common Pipeline Errors & Root Causes

          

             

                
1. Repository Checkout Timeout (120s)

                
Meaning: The pipeline could not download your code.
Root Cause: This is almost always an authentication issue. If deploying a private repository, ensure you have provided a valid GitHub Personal Access Token. Without it, Git prompts for a password in the background and waits until the pipeline times out.

             

             
             

                
2. Port Allocation Conflicts (e.g. Connection Refused / Port in Use)

                
Meaning: Docker failed to start the deployment container because the required port is already taken.
Root Cause: Usually caused by hardcoded port configurations in docker-compose files. Ensure you only expose the absolutely necessary ports. The AI-CICD-Monitor handles dynamic port routing automatically to prevent these collisions.

             

             
             

                
3. Health Check: Connection Refused

                
Meaning: The container is running perfectly fine, but the URL pinged by the monitor timed out or was refused.
Root Cause: This is usually a mismatched port configuration. For example, if you deploy a static website using Nginx, Nginx binds to port 80 inside the container. If the health checker incorrectly probes port 8100 inside the container, the connection is refused, resulting in a 'Deployed with Warnings' status.

             

             
             

                
4. Dashboard Stuck on "Loading..."

                
Meaning: The frontend dashboard couldn't load or update the latest logs from the Python backend.
Root Cause: Usually caused by aggressive browser caching of API endpoints, or a JavaScript crash. It can also occur if you modify the UI code but forget to rebuild the backend docker container (docker compose up -d --build) to serve the new HTML.

📄 Writing your deployment.txt

          
          

            The AI-CICD-Monitor uses a strict, language-agnostic deployment configuration. 
            It does not guess your framework or language. You must explicitly tell the engine exactly how to build and start your application using deployment.txt.
          


          
Required Variables

          
    
             
  • project_name: A unique, URL-safe name for your application (e.g., project_name=my-app).
    • 
          


          
Optional Variables & Special Cases

          
    
             
  • ssl: Set to true to automatically provision a Let's Encrypt certificate via Nginx.
    • 
             
  • domain: A custom domain (e.g., domain=api.myapp.com). If omitted, a wildcard subdomain is generated automatically.
    • 
             
  • health_check: The HTTP path to verify success (e.g., health_check=/api/health). Defaults to /.
    • 
             
  • rollback_enabled: Set to true to automatically revert the symlink if the deployment fails health checks.
    • 
             
  • working_directory: Used in Monorepo blocks (e.g., working_directory=frontend) to specify where the build/start commands execute.
    • 
          


          
Multi-Line Execution Blocks

          

             Instead of cramming long shell scripts into a single line, use Markdown Headers and Codeblocks to write your build and start scripts naturally. The engine executes these exactly as written using the dedicated deployrunner system user.
          


          

project_name=my-node-app

port=3000

ssl=true
